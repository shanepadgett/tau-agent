Tau Agent Context Harness Research Report

Research into lower-cost, higher-reliability coding agents. The work started with read/edit tools and expanded into state, context projection, validators, prompt caching, code intelligence, and recovery.

The transcript is the wrong place to store working evidence.

Coding agents keep paying for old search results, stale file snapshots, passing logs, and errors that have already been fixed. The useful pattern is a managed harness: store exact evidence locally, keep handles for recovery, and compile a small current prompt view for each model turn.

What this report covers

Scope is research-backed agent runtime design for Tau: what we read, what we tested, what held up, and where the uncertainty remains.

  • Benchmark corpus Primary repos were pi, codex, and opencode. tau-agent stayed a small control repo.
  • Reference designs We compared patch tools, line-addressed reads, deterministic conversation compaction, reference lookup journals, and command-aware output compression.
  • Research artifacts Simulations covered read policy, context packs, turn cost, prompt caching, compaction, edit formats, validators, code intelligence, subagents, JSON shape, schema shape, and retry behavior.

What we investigated.

The original question was smaller: could Tau get better read, search, and modification tools? The answer was yes. The bigger finding was that file tools alone cannot fix the waste. Reads, searches, edits, validators, probes, LSP, and subagents all need the same lifecycle model.

File access and edits

Compared whole-file reads, range reads, exact string edits, line/range operations, hidden handles, visible hashlines, batch writes, stale digest failures, and LSP workspace edits.

Search and scope

Modeled search/read tradeoffs, small owned-file reads, scoped context packs, and the cost of broad packs carrying unrelated material.

Runtime state

Studied deterministic conversation compaction and reference lookup so old evidence can be retired without losing exact recovery.

Tool output

Tested validators, shell output reducers, compact diagnostic rows, arbitrary JSON payloads, hidden raw handles, and passing-log suppression.

Provider behavior

Reviewed Anthropic, OpenAI, Gemini, and xAI cache behavior. Stable prompt prefixes and late runtime capsules matter more than clever per-turn prompt rewriting.

Agent workflow

Modeled deterministic scouts, typed subagent patches, durable preference detection, restricted-shell modes, controlled probes, and code-intelligence tools.

How ordinary tool calls turn into context debt.

Current agents often treat the chat transcript as the database. That works for a short demo. It gets expensive and confusing during real repo work because the prompt keeps carrying material that is no longer the best evidence.

search
A repo search returns dozens or hundreds of lines. Later reads make most of those hits obsolete, but the old hits stay visible.
read
A file read gives the model useful context. After an edit, that read becomes stale unless the harness retires it and returns a refreshed snapshot.
check
A passing command can print thousands of tokens. With a passing result, those tokens have no decision value, but they keep renting prompt space.
retry
The model may reason over old failures, ask for redundant reads, or patch around an error that was already resolved.

The core model: event log to state to prompt view.

The useful architecture is a small runtime that records tool events, keeps exact artifacts off to the side, and decides which compact facts belong in the current prompt view.

Tool event log

Every read, search, edit, check, probe, LSP action, and subagent result becomes a structured record with path, type, turn, digest, visible size, hidden reference, and status.

Local state store

Exact file bytes, full logs, raw JSON, search result sets, and diagnostics stay local. The prompt gets handles, selected lines, summaries, and unresolved facts.

State to view compiler

Deterministic rules mark records active, pinned, covered, superseded, stale, resolved, or failed. Covered search results and stale snapshots leave the visible prompt.

Compiled prompt view

The model sees current file capsules, active user constraints, unresolved failures, budget status, and handles it can expand when exact detail is needed.

visible capsule hidden exact snapshot: @f3 
@f3 src/user.ts lines 35..58 digest=9c31e2
35| export function getName(user: User) {
36|   if (!user.profile) return "unknown";
37|   return user.name;

Requested change: line 37 should return user.profile.displayName.
Write rule: disk changes require a matching @f3 digest; successful writes return a refreshed snapshot.

Evidence summary.

The simulations were break-even checks. They were useful because they showed where simpler approaches still win, where compactness breaks quality, and where a managed context layer has real leverage.

Recurring context cost.

Context cost has two parts: the initial tool result plus every future turn that keeps that result visible. Stale visible output also competes for model attention and can cause retries.

The useful accounting model

Visible artifacts cost model attention. Hidden artifacts cost local storage and occasional lookup. A harness should make that tradeoff visible instead of letting raw tool output become permanent prompt history.

recurring context cost ≈ visible tokens × future turns × retry pressure
Artifact Keep visible? Reason
Current file lines Yes, while active. The model needs exact nearby code to edit safely.
Old search hits covered by file read No. The file snapshot is better evidence. Search hits can stay hidden for audit.
Passing build log No. Status matters; raw log rarely helps for passed commands.
Unresolved failure Yes, compactly. The model needs the diagnostic until a passing check resolves it.
Hidden handle Only as a pointer. A handle is recoverable evidence, not evidence by itself. Expand it for exact reliance.

Candidate capabilities and confidence.

The buildable surface is larger than a new read tool. The common thread is controlled evidence: tools produce records, records update state, and the prompt view stays small enough for the model to use.

Capability Problem solved How it would work Evidence / confidence
Read tool Raw reads are either too small and turn-expensive or too large and sticky. Whole-read small owned files; range-read large files; store exact bytes, line map, digest, token estimate, mtime, and handle. Strong. Repo-size and turn-cost sims support the threshold policy.
Search tool Search dumps duplicate themselves and become stale when reads cover them. Group hits by file/symbol/range, hide full result set, auto-read small owned files when it avoids a turn, retire hits covered by reads. Strong for controlled repos; needs real task measurement for auto-read aggression.
Batch modification tool Multi-file edits cost turns and stale snapshots cause bad patches. Accept line/range replace, insert, delete, create, move, delete-file ops; preflight paths, digests, conflicts, and overlap; refresh capsules on successful write. Strong for text edits. Fuzzy relocation should wait; strict stale failure is safer.
Runtime state store Chat history cannot tell what is current, stale, hidden, resolved, or superseded. Store records with lifecycle states such as active, pinned, covered, superseded, stale, hidden, resolved, and failed. Central finding. Needed by reads, edits, validators, probes, LSP, subagents, and compaction.
State to view compiler Ad hoc compaction loses exact facts or keeps obsolete material. Compile a deterministic prompt view from state for each model turn. Keep active constraints and unresolved failures; hide covered bulk. Strong. Projection and capsule-lifetime sims showed the largest savings with quality preserved.
Context budget tool Users and agents cannot see why prompts are large. Report active visible tokens by capsule, hidden bytes by artifact, stale handles, unresolved failures, and prune candidates. High confidence. Also needed for real cost reports.
Quiet validators Build/test/typecheck output is useful when failing and wasteful when passing. On pass, return status and hidden log handle. On fail, extract compact diagnostics. Retire failure capsules when a passing check resolves them. Strong. Validator and output-compression sims favored quiet status plus compact failures.
Prompt-cache-aware layout Runtime posture and tool churn can break provider prompt caches. Keep stable instructions and tool schemas early. Put volatile runtime state late. Avoid rewriting the system prompt for every posture change. Strong direction from provider research; actual Pi request capture still required.
Compact JSON and row formats Repeated object JSON and pretty output burn tokens without adding information. Use tuple/legend or interned rows for repeated diagnostics and search hits. Store raw JSON hidden; show compact projections. Good simulation support. Needs model-behavior testing for readability and error rate.
LSP code intelligence Text edits are weak for rename, imports, references, and diagnostics. Use LSP for semantic rename/import/reference work when warm; fail over to text/search when unavailable. Promising. Cold-start/index cost still unknown.
Controlled probes Agents use shell commands to answer questions that could be safer, smaller probes. Provide restricted read-only probes for package metadata, symbols, project config, dependency graph, and generated summaries. Medium. Needs security defaults and real workflow measurement.
Restricted-shell mode Shell is powerful but noisy, unsafe, and hard to summarize correctly. Prefer typed tools for read/search/check/probe. Allow shell only when structured tools cannot answer. Medium. Output compression research supports it, but full lockdown needs UX testing.
Deterministic scouts Broad subagents can cost more than they save. Use focused local discovery agents that return typed findings or context patches, not prose essays. Strong for local discovery in sims; web and broad synthesis can still justify richer subagents.
Preference capture Durable user preferences and reversals get buried in chat. Use cheap prompt-only classification for high-impact preferences; require confirmation for durable reversals. Promising but lower importance than the core state and evidence lifecycle. Needs behavior testing.

Expected operational impact.

Token savings matter, but the bigger value is control. A harness can reduce spend, reduce retries, make edits safer, and give every visible fact a stored source.

What gets better

  • Smaller active prompts because old evidence leaves the compiled view.
  • Fewer redundant discovery turns because small relevant files can be read up front.
  • Safer edits because handles and digests gate disk writes.
  • Cleaner validation because passing logs stop occupying prompt space.
  • Better auditability because visible claims can point back to hidden exact artifacts.
  • Better enterprise controls because path policy, shell policy, secret handling, and budget reporting become explicit.

Recommended system properties.

The recommendation is the whole harness pattern: exact evidence stays recoverable, the visible prompt stays current, and every tool result has a lifecycle.

State lifecycle

  • Tool events create records with path, turn, digest, visible size, hidden reference, and status.
  • Records move through active, covered, superseded, stale, resolved, failed, and pinned states.
  • The compiled prompt view keeps active constraints and unresolved failures while retiring covered search/read/check output.

File and edit safety

  • Reads and searches return compact visible capsules plus hidden exact artifacts.
  • Batch modifications use handle, digest, path, overlap, and conflict checks to gate disk changes.
  • Successful writes refresh snapshots immediately; stale handles fail safely without disk changes.

Context accounting

  • Budget reports show active visible tokens, hidden artifacts, stale handles, unresolved failures, and prune candidates.
  • Each visible capsule explains why it remains in the prompt.
  • Cost reports compare raw transcript behavior against the compiled prompt view on real tasks.

Validation behavior

  • Passing checks return status and hidden log handles.
  • Failing checks return compact diagnostics that remain visible until resolved.
  • Checks that modify files refresh affected snapshots and digests.

Open questions.

The remaining gaps need measured sessions against a working harness, not more broad spreadsheet runs. These are the places where real model behavior or provider plumbing could change the design.

Provider request capture

Inspect actual Pi payloads: where tools land, what changes bust cache, whether runtime state can sit late, and whether cached/uncached token accounting is available.

Compact format behavior

Test whether models reliably use tuple rows, interned diagnostics, compact JSON projections, file capsules, and keep/ref/drop anchors without raising error rate.

Hidden lookup discipline

Validate that models treat handles as pointers. Visible capsule text is evidence; hidden content must be expanded for exact reliance.

LSP cold/warm cost

Measure TypeScript server startup and operation cost on pi, codex, opencode, and tau-agent. LSP eagerness depends on those timings.

Validator scope

Compare dirty-scope checks with full checks. The harness should run enough validation to catch mistakes without turning every task into a full build.

Security boundaries

Hidden logs can contain secrets. Files, logs, shell output, and web text are data. Probes should default to stripped environment and no network.
